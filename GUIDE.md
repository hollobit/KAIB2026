# Machine-Readable AI 재정계획 문서 작성 가이드라인

> 기획재정부 → 각 부처 예산 편성 시 참고용
> 목적: AI 재정사업 예산서의 기계 판독성(machine-readability) 확보

---

## 1. 개요

### 1.1 배경

현행 AI 재정사업 예산서(PDF)를 구조화된 데이터로 변환하는 과정에서, 양식의 비표준화·비일관성으로 인해 533개 사업 중 상당수에서 자동 추출 오류가 발생하였다. 본 가이드라인은 부처가 예산서를 작성할 때 기계 판독이 가능한 수준의 일관성을 확보하기 위한 **요구사항(MUST)**과 **권고사항(SHOULD)**을 제시한다.

### 1.2 법적 근거 및 관련 제도

- **국가재정법 제9조**(재정정보의 공표): 정부는 예산·기금의 주요 내용을 공표해야 하며, 본 가이드라인의 구조화 데이터 병행 배포는 이 조항의 취지를 구체화한다.
- **공공데이터의 제공 및 이용 활성화에 관한 법률**(공공데이터법): 공공기관은 기계 판독이 가능한 형태로 데이터를 제공해야 한다(제3조, 제22조).
- **dBrain(디지털예산회계시스템)**: 부처는 이미 dBrain을 통해 예산요구서를 구조화된 데이터로 입력하고 있으며(국가재정법 제31조), 본 가이드라인은 이 데이터가 PDF 변환 과정에서 손실되지 않도록 양식을 표준화하고, 나아가 dBrain 데이터의 직접 공개를 지향한다.
- **예산안편성지침**(국가재정법 제29조, 매년 3.31 통보): 본 가이드라인은 편성지침 부속 자료로 배포하여 부처의 예산서 작성 단계에서 적용되도록 하는 것을 권고한다.

### 1.3 적용 범위

- AI 재정사업 전체 (R&D, 정보화, 일반)
- 예산 총괄표, 기능별(내역사업별) 테이블, 사업설명자료

### 1.4 용어 정의

| 용어 | 정의 |
|------|------|
| MUST (필수) | 반드시 준수해야 하는 요구사항. 미준수 시 기계 판독 불가 |
| SHOULD (권고) | 준수를 강력 권장하는 사항. 미준수 시 판독 정확도 저하 |
| MAY (선택) | 준수 시 분석 품질 향상에 기여하는 사항 |

## 2. 문서 구조 요구사항

### 2.1 섹션 헤더 규격

#### 요구사항 (MUST)

각 사업의 예산서는 아래 섹션을 **정확한 순서와 헤더 문구**로 포함해야 한다.

```
가. 예산 총괄표
□기능별(내역사업별) 예산안 내역
  ○기능별 분류(합계)
나. 사업설명자료
  1) 사업개요
  2) 사업목적
  3) 사업내용
  4) 법적근거
  5) 사업관리자
  6) 예산 산출근거
  7) 사업 집행절차
```

**금지 변형** (현재 발견된 문제 사례):

| 현행 변형 (금지) | 표준 형태 (필수) |
|-----------------|-----------------|
| `□ 예산 총괄표` | `가. 예산 총괄표` |
| `□기능별(내역사업별), 목별 예산안 내역` | `□기능별(내역사업별) 예산안 내역` |
| `□기능별(내역사업별), 세사업별 예산안 내역` | `□기능별(내역사업별) 예산안 내역` |

#### 권고사항 (SHOULD)

- 목별 예산이 필요한 경우, `□기능별(내역사업별) 예산안 내역` 다음에 별도 섹션 `□목별 예산안 내역`으로 분리
- 각 섹션 헤더는 해당 줄에 단독으로 위치 (같은 줄에 다른 텍스트 혼재 금지)

### 2.2 테이블 구조 규격

#### 2.2.1 예산 총괄표 (MUST)

7컬럼 고정 구조를 사용한다(구분 열 제외, 데이터 컬럼 기준). 추경 미편성 시에도 7컬럼을 유지하고 추경 컬럼에 `-`를 표기한다.

```
| 구분 | 2024 결산 | 2025 본예산(A) | 2025 추경(B) | 2026 요구 | 2026 확정예산(C) | 증감(C-A) | 증감률(%) |
```

> **용어 주의**: 여기서 "2025 본예산"은 정기국회에서 의결된 당초예산(본예산)을 의미하며, "2025 추경"은 추가경정예산이 편성된 경우의 수정예산현액을 의미한다. 추경 미편성 시 추경 = 본예산이다. "2026 확정예산"은 국회 의결을 거쳐 확정된 세출예산을 의미한다. 증감액(C-A)은 2026 확정예산과 전년도 본예산의 차이로 산출한다.

**세부 규칙**:

| 항목 | 요구사항 |
|------|---------|
| 컬럼 수 | **7컬럼 고정** (MUST, 구분 열 제외). 추경 미편성 시 추경 컬럼에 `-` 표기 |
| 단위 | `백만원` 고정 (MUST). 표 상단에 `(단위: 백만원)` 명기 |
| 음수 표현 | `△` 통일 (MUST). `▵`(U+25B5), `-`, 괄호 등 혼용 금지 |
| 순증/순감 | 증감률 컬럼에 `순증` 또는 `순감` 텍스트 표기 (MUST) |
| 빈값 | `-` 통일 (MUST). 공백, `0`, `—` 혼용 금지 |
| 사업코드 | 테이블 외부에 별도 행으로 표기 (MUST). 테이블 내부에 `(1134-309)` 삽입 금지 |
| 숫자 서식 | 천단위 쉼표 필수 (MUST). `1,234,567` |

#### 2.2.2 기능별(내역사업별) 테이블 (MUST)

11컬럼 고정 구조를 사용한다.

```
| 구분 | 2024 예산(1) | 2024 예산현액(2) | 2024 집행(3) | 2024 이월(4) | 2024 불용(5) |
|      | 2025 예산(6) | 2025 예산현액(7) | 2025 집행(8) | 2025 이월(9) | 2025 불용(10) | 2026(11) |
```

**세부 규칙**:

| 항목 | 요구사항 |
|------|---------|
| 첫 행 | `○기능별 분류(합계)` 또는 `○기능별분류(합계)` (MUST) |
| 내역사업 접두사 | `․` (U+2024, one dot leader) 통일 (MUST). `ㆍ`(U+318D) 금지 |
| 하위분류 | `- 인건비`, `- 국고보조` 등 하위 분류 행은 **테이블에 포함하지 않음** (MUST) |
| 합계 행 | 내역사업 합계 = 총괄표 2026 예산과 반드시 일치 (MUST) |
| 브래킷 값 | `[107448.5]` 등 참조 값은 테이블 내에 삽입하지 않음 (SHOULD) |
| 페이지 번호 | `- NNN -` 형태의 페이지 번호가 테이블 데이터 사이에 위치하지 않도록 함 (SHOULD) |

#### 2.2.3 원문자(①②③) 항목 (SHOULD)

- 기능별 테이블에서 원문자 접두사 항목을 사용하는 경우, **하위 dot(․) 항목이 반드시 따라와야 함**
- 하위 항목 없이 원문자만 사용하는 경우 → dot(․) 접두사로 변경

### 2.3 사업 식별 정보 (MUST)

각 사업은 다음 필드를 구조화하여 포함해야 한다.

```
사업코드: NNNN-NNN
사업명: (전체, 잘림 없이. R&D/정보화 접미사 포함)
소관부처:
소관부서:
시행기관:
회계유형: 일반회계 | 특별회계 | 기금
사업상태: 계속 | 신규
지원유형: 직접 | 출연 | 보조 | 융자 | 출자 | 민간보조
사업기간: YYYY년~YYYY년(N년)
총사업비: N,NNN백만원 (국비: N,NNN)
```

**사업명 규칙** (MUST):
- 전체 사업명을 **한 줄에 완전히** 표기 (줄바꿈/잘림 금지)
- R&D 사업: 사업명 끝에 `(R&D)` 필수 부착
- 정보화 사업: 사업명 끝에 `(정보화)` 필수 부착
- `(R&D,정보화)` 중복 유형도 명시적 표기

## 3. 데이터 교환 형식 요구사항

### 3.1 구조화 데이터 병행 배포 (MUST)

PDF 예산서와 함께 **JSON 형식의 구조화 데이터 파일**을 공식 배포한다. 부처는 이미 dBrain에 구조화된 예산 데이터를 입력하고 있으므로, dBrain 데이터를 JSON 스키마에 맞추어 자동 변환·추출하는 방식으로 부처의 추가 업무 부담을 최소화한다. 배포 경로로는 열린재정(fiscal.go.kr) 또는 공공데이터포털(data.go.kr) Open API를 활용할 수 있다(공공데이터법 제22조).

### 5.1 자동 검증 항목 (MUST)

부처가 dBrain을 통해 예산요구서를 제출할 때, 다음 항목을 dBrain 내에서 자동으로 검증해야 한다. 현재 dBrain에도 일부 정합성 검증이 내장되어 있으나, 아래 규칙을 추가 적용하여 PDF 변환 후에도 데이터 품질을 보장한다.

| # | 검증 규칙 | 허용 오차 |
|---|----------|----------|
| 1 | 내역사업 예산 합계 = 총괄표 2026 예산 | ±1% |
| 2 | 증감액 = 2026 본예산 - 2025 본예산(A) | 정확 일치 |
| 3 | 증감률 = 증감액 / 2025 본예산 × 100 | ±0.1%p |
| 4 | 모든 금액 필드는 0 이상의 수(number) (음수는 증감액·증감률만 허용) | - |
| 5 | 사업코드 형식: `NNNN-NNN` | 정규식 매칭 |
| 6 | 사업명이 50자 이내이면 잘림 없이 완전 표기 | - |
| 7 | R&D 사업은 `(R&D)` 접미사 필수 | - |
| 8 | 신규 사업의 2024/2025 예산은 null 또는 0 | - |
| 9 | 추경 미편성 시 `2025_supplementary` = `2025_original` 또는 null | 정확 일치 |
| 10 | `change_rate` = `change_amount` / `2025_original` × 100 (본예산 기준 증감률) | ±0.1%p |
| 11 | 기능별 테이블의 불용액(5,10열) + 이월액(4,9열) + 집행액(3,8열) ≤ 예산현액(2,7열) | 정확 일치 |
| 12 | 기능별 테이블의 예산현액 = 예산 + 추경 + 전년도 이월 (예산현액 구성 검증) | ±1% |

### 5.2 교차 검증 항목 (SHOULD)

기존 유사·중복사업 정비(기재부 예산실) 및 재정사업자율평가(국가재정법 제8조의2)를 보완하여, dBrain에서 예산요구서 제출 시점(매년 5.31)에 자동 수행한다.

| # | 검증 규칙 | 목적 | 기존 제도 연계 |
|---|----------|------|--------------|
| 1 | 동일 사업코드 타부처 중복 확인 | 중복 편성 방지 | 유사·중복사업 정비 |
| 2 | 사업명 유사도 40% 이상 기존 사업과 비교 | 유사사업 사전 식별 | 유사·중복사업 정비 |
| 3 | 동일 시행기관의 타부처 사업 목록 확인 | 교차부처 조정 | 재정사업자율평가 |
| 4 | 2025 대비 +50% 이상 증감 사유 기재 확인 | 이상치 설명 | 기재부 사정 |
| 5 | 전년도 자율평가 '미흡' 판정 사업의 예산 증액 시 사유 확인 | 성과 연계 | 재정사업자율평가 |

## 6. PDF 양식 호환성 요구사항

### 6.1 텍스트 인코딩 (MUST)

- 문서 전체 UTF-8 인코딩
- 특수 문자 표준화:

| 용도 | 사용할 문자 | 금지 문자 |
|------|-----------|----------|
| 내역사업 접두사 | `․` (U+2024) | `ㆍ`(U+318D), `·`(U+00B7), `•`(U+2022) |
| 음수 기호 | `△` (U+25B3) | `▵`(U+25B5), `▲`(U+25B2), `-`(하이픈) |
| 가운뎃점 | `·` (U+00B7) | `ㆍ`(U+318D), `∙`(U+2219) |

### 6.2 테이블 레이아웃 (MUST)

- 테이블 셀 경계가 PDF 텍스트 추출 시 명확히 구분되어야 함
- **셀 병합 최소화**: 특히 데이터 행에서 셀 병합 금지
- **페이지 경계 처리**: 테이블이 페이지를 넘어가는 경우, 다음 페이지에 컬럼 헤더 반복

### 6.3 사업명 표기 (MUST)

- 사업명은 **절대로 줄바꿈하거나 잘라서 표기하지 않음**
- 사업명이 긴 경우에도 전체를 한 줄에 표기
- `(R&D)`, `(정보화)`, `(R&D,정보화)` 접미사가 잘리지 않도록 보장

## 7. 이행 로드맵

> **실현 가능성 참고**: 예산서 양식 변경은 예산안편성지침(국가재정법 제29조, 매년 3.31 통보)에 반영하여 차년도 예산 편성부터 적용할 수 있다. dBrain 시스템 개선은 기획재정부 정보화사업으로 예산을 확보하여 추진한다. 각 Phase의 시점은 예산안편성지침 반영 시기(T+1년)를 기준으로 한다.

### Phase 1 (즉시 ~ 차년도 편성지침 반영): PDF 양식 표준화
- 예산안편성지침 별첨으로 본 가이드라인 배포
- 섹션 헤더 통일, 특수문자 표준화, 사업명 잘림 방지
- dBrain에서 PDF 생성 시 정합성 검증 규칙 추가 (내역사업 합계 = 총예산)
- **기대 효과**: 부처 추가 업무 부담 최소 (기존 양식의 소규모 표준화)

### Phase 2 (6개월 ~ 1년): dBrain 데이터 기반 JSON 병행 배포
- 사업별 JSON 스키마 확정 (본 가이드라인 3.1절)
- **dBrain에서 JSON 자동 추출·변환 기능 개발** (부처가 별도 JSON을 작성하는 것이 아니라, dBrain 데이터를 자동 변환)
- 열린재정(fiscal.go.kr) 또는 공공데이터포털(data.go.kr)을 통한 Open API 배포
- 자동 정합성 검증 시스템 구축 (PDF와 JSON 간 교차 검증 포함)
- **기대 효과**: 부처 추가 업무 없이 구조화 데이터 공개

### Phase 3 (1년 ~ 2년): AI 사업 분류 체계 도입
- dBrain에 AI 기술 유형(9종)·R&D 단계(5종)·AI 도메인 태깅 필드 추가
- 예산요구서 제출(5.31) 시 태깅 입력 의무화
- dBrain 내 교차부처 유사사업 자동 탐지 기능 구현 (기존 유사·중복사업 정비 프로세스와 연계)
- **유의사항**: dBrain 필드 추가는 시스템 개편이 필요하므로 기재부 정보화예산 확보 선행 필요

### Phase 4 (2년 ~ 3년): 통합 데이터 플랫폼
- dBrain·NTIS·e-나라도움 간 AI 사업 데이터 연계 뷰 구현
- 국회 예산정책처(NABO) 분석 시스템과 dBrain API 연동
- 공개 API 기반 데이터 배포 (시민사회·연구기관 활용)
- **유의사항**: 시스템 간 연계는 부처 간 협의·보안 심사가 필요하여 장기 추진 과제
